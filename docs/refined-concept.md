# Loom 提炼方案（Refined Concept）

> 这是综合 4 视角评估 + 4 方向调研后，对原始想法的"提升版"。保留精华、修正薄弱点、明确差异化、给出 MVP 边界与推荐架构。

## 提炼后的定位

把 Loom 从"组装整个 GitHub"**收敛为"受控生态内的配方式组装平台"**：

> 在单一锁定生态（v1 = Node/TS Web）内，把**策展过**的真实开源项目离线规范化为带接口契约、依赖足迹与 license 标注的 **Project Capsule**。用户输入想法后，沿**预定义接缝(seam)** 声明式装配一个连贯的 core starter + 少量自包含模块（auth/支付/上传）。AI 的主战场不是生成部件，而是**生成把部件粘起来的 adapter/胶水层**，最终以最薄的 **AST 语义 diff** 落地。

真正可防御的不是想法空间 UI 漏斗，而是随使用累积的**"组装兼容性知识"**——每次组装是否编译/测试/上线成功的反馈数据。

**卖点应从"省 token"换成"更快交付经过验证的可运行代码"；token 节省是副产品而非主张。**

## 差异化与护城河（命门同一点）

新颖性 = 两件互锁的事：
1. **范式倒置**：默认从`生成-再修复`(Copilot/v0/bolt) 改为`检索-组装-再 diff`，把生成降级为例外。
2. **可组装性层是唯一可防御资产**：把原始 repo 规范化为接口类型化、带依赖图与 license 的可组装单元，并通过**组装成功率反馈闭环**形成随使用复利、难被复刻的数据护城河。

> 命门和护城河是同一个点：**组装兼容性知识**，而非检索或生成本身。要诚实——把 adapter/胶水生成做成核心能力，而不是假装"最小 diff"自动成立。

## 推荐架构（重心：从"相似度"转向"接缝建模"）

```
┌─ 1. 想法空间层（语义层）─────────────────────────────┐
│  分面分类 + 涌现式聚类（替代固定树）                  │
│  · 结构化分面：领域 / 应用原型 / 技术栈 / 能力标签    │
│  · 顶层大类（如"网站"）仅作粗粒度 facet              │
│  · 细分类目由语料聚类自动派生，避免分类漂移          │
│  · 增加"想法 → 能力清单"分解层，对齐组装粒度          │
└──────────────────────────────────────────────────────┘
┌─ 2. 索引层（Project Capsule）── 分层多策略 ──────────┐
│  L0 仓库元数据（语言/框架/license/构建系统，硬过滤） │
│  L1 模块/能力目录（AST+import 图抽边界、接口、依赖） │
│  L2 多粒度 embedding（项目/模块/能力级，非整库 blob）│
│  L3 符号/调用图（供抽取与 diff）                      │
│  检索 = 标签硬过滤 → 三信号(向量+AST+依赖图)排序      │
│         → 可组合性校验                                │
│  内容寻址（借鉴 Unison）给片段稳定标识，降版本漂移   │
└──────────────────────────────────────────────────────┘
┌─ 3. 配方(manifest) 与 Composition IR 分离 ───────────┐
│  manifest 声明意图，字段：                            │
│    · base/core   选定骨架，按 commit hash pin 死      │
│    · grafts      各模块 source@ref + 选择器 + 目标路径│
│    · bindings/adapters  接缝调和（一等公民，非事后）  │
│    · overrides   最薄人/AI 补丁层                     │
│    · lock/provenance  每源内容哈希 + license          │
│  确定性 resolver 把 manifest 编译成 IR 再物化文件树   │
│  core→grafts→adapters→diff 每层纯变换、可重放          │
│  （同构于 Kustomize / Nix overlay）                   │
└──────────────────────────────────────────────────────┘
┌─ 4. 物化与补丁层（diff-on-assembly）─────────────────┐
│  diff 针对物化结果生成                                │
│  用 AST 级语义补丁 或"意图+再生成"兜底（非行号 patch）│
│  常见接缝失配做成可复用 adapter 目录条目              │
└──────────────────────────────────────────────────────┘

横切：License 合规闸门 · golden 集成测试 · token/跑通率度量回路
借鉴：aider 的 repo-map+多 edit 格式 · Nx Tree 可预览回滚 · Plop/Hygen 的 inject
```

## MVP 边界

**做：**
- 锁定单一生态与构建链（v1 = Node/TS Web），拒绝跨语言/跨 bundler 组装。
- 离线 ingestion 管线：把策展的少量高质量项目编译成统一 Project Capsule（manifest + 模块目录 + 接口契约 + 依赖足迹 + SPDX license + 多粒度 embedding）；索引只面向 Capsule。
- 默认"整套复用单一连贯 starter"作为 core；多源组装只限接口干净、自包含的模块（auth/支付/上传），**禁止跨源拼内核逻辑**。
- License 合规闸门前置（SPDX + 兼容矩阵 + ScanCode），默认排除 copyleft。
- 重写阈值 + 适配比（diff/复用 > 0.4）实时止损，超阈值自动回退从零生成。
- 每个跨源接缝显式生成 adapter + 接缝测试；每个候选跑 golden 集成测试，"能否跑通"作唯一可组合性信号。
- 埋点度量回路：diff 体积 / 一次跑通率 / 含适配的实际总 token。

**不做：**
- 任意 GitHub repo 自由拼接；对单体做程序切片 feature location（规模上不可靠）。
- 跨语言/跨 bundler 组装；"网站"这种大类一上来就泛化。
- 把"最小化 token"当头号卖点；把"多源异源 core"当默认推荐路径。

## 杀手级场景（收益最大）

1. **同质化垂直域的连贯 starter 复用**：营销站/落地页、CRUD SaaS 后台——结构一致、接缝少，整套复用 + 薄 diff 净赢面最大。
2. **自包含横切模块的受控注册表组装（shadcn 模型）**：auth/支付/上传/通知，沿强制 adapter 契约嵌入 core。
3. **样板密集骨架生成**：Web 脚手架 + 构建/配置 + 基础 RBAC，output token 大且标准化。
4. **"经验证 golden path"内部平台（对标 Backstage）**：组织内策展的高质量项目作可信组装源，反馈在封闭语料上快速复利。

## 待解问题（开工前要回答）

1. 锁定单一生态后，可策展的高质量、接口干净的源项目池有多大？是否够支撑"想法→匹配"召回率，还是频繁触发重写回退、退化成又一个 codegen？
2. 组装兼容性反馈闭环需多少真实样本才能让匹配排序显著优于纯相似度？冷启动期如何避免数据稀疏导致护城河无法启动？
3. "理解 token"在哪个复用比/自包含度下才真正被 output token 节省覆盖？需要离线基准用数据划定每个垂直域盈亏边界。
4. adapter/胶水层能多大程度自动生成并通过接缝测试？若 adapter 本身需大量人工/高成本调试，组装相对"从零连贯生成"的优势是否还成立？
5. 定价/价值主张如何对冲"推理成本下降+上下文窗口扩大"对 token 卖点的侵蚀？能否把"可追溯+合规+经验证可运行"做成信任卖点？
6. 内容寻址 pin 死源 commit 后，上游 CVE 如何回传修复？被组装项目的供应链安全与持续维护责任如何界定？

## 下一步建议

先做**离线基准实验**回答问题 1 和 3：在单一垂直域（如营销站）选 5–10 个策展项目，对同类任务分别跑"组装"与"从零生成"，用 `总 token / 一次跑通率 / 适配比` 三个数据决定要不要继续投入。**用数据验证核心假设，而不是把"省 token"当信仰。**
